# Always 项目详解

[English](PROJECT_DETAILS.md) | [中文](项目详解.md) | [返回 README](README.zh.md)

> 这份文档说明 Always 的运行机制、能力边界、取舍、风险和维护流程。面向普通用户的安装与命令说明见 README。

## 一、第一次接触：它是什么，为什么需要它

Always 是一个本地可复用提示词 / 常用句系统，用来保存会反复交给 AI Agent 的指令，并通过 Raycast、Codex、Claude Code 和 CLI 读取同一份内容。

**它解决的问题**：真正好用的指令经常重复。比如先规划再写代码、审查某个文件的特定风险、保持某种写作口吻、完成前必须验证。每次重写既浪费时间，也容易让要求逐渐走样。

**核心路径**：

```text
保存一次 -> 快速找到 -> 填写变量 -> 粘贴 -> 检查 -> 自己发送
```

**为什么采用这种形态**：

- 个人指令应该可读、可编辑、可迁移。
- Raycast、Codex 和 Claude Code 不应该维护几套逐渐漂移的提示词。
- 选择常用句应该比翻笔记软件或等待 Agent 对话往返更快。
- 自动粘贴不等于自动执行，最终发送权必须属于用户。
- 不依赖云服务、API Key、数据库服务或第三方 Python 包。

**它不是什么**：

- 不是项目记忆或任务交接日志；项目级跨 Agent 状态应该交给 ShareMemory。
- 不是文档管理器；条目是文字指令，不是附件。
- 不是密码库；所有数据都是明文 JSON。
- 不是提示词商城或跨设备同步服务。
- 不是完整的剪贴板历史工具。

## 二、它具体怎么运作

安装后的系统分成四部分：

```text
Always 源码仓
  scripts/install.sh
  scripts/raycast/
    always.sh                 Raycast 原生选择器启动器
    open-json.sh              Raycast 打开真实 JSON 文件
    _always_common.zsh        Raycast 脚本共享辅助函数
  skills/always/
    SKILL.md                  Agent 触发与执行规则
    scripts/always.py        CLI、选择器、变量渲染、持久化
    assets/sentences.sample.json

Raycast 入口
  把 <repo>/scripts/raycast 添加为 Script Directory
  把 Always 绑定为全局快捷键，通常推荐双击 Option（⌥⌥）

Agent 发现路径
  ~/.agents/skills/always    -> 源码仓（Codex）
  ~/.claude/skills/always    -> 源码仓（Claude Code）

个人运行数据
  ~/.always/sentences.json
  ~/.always/sentences.backup.json
```

### 2.1 安装模型

`scripts/install.sh` 创建的是软链接，不是复制件。两个 Agent 因此执行相同的 `SKILL.md` 和 `always.py`。更新源码仓后，两个 Agent 集成立即使用新代码。

只有在 `~/.always/sentences.json` 不存在时，安装器才会从 sample 初始化数据库。重复安装不会重置个人内容。

Raycast 集成有意保持显式配置，而不是自动写入设置：用户需要把 `<repo>/scripts/raycast` 添加为 Raycast Script Directory，并在 Raycast Settings 里分配快捷键。安装器只打印目录路径，不修改 Raycast 偏好设置。

### 2.2 选择模型

核心选择器是 `always.py menu`。Skill 默认走它，因为它能减少一次 Agent 对话往返；Raycast 则让这个选择器不需要打开 Agent 对话也能全局调用：

1. 读取 JSON 数据库。
2. 应用可选搜索词。
3. 用 AppleScript 展示匹配标题。
4. 通过原生弹窗填写 `{变量}`。
5. 用 `pbcopy` 复制渲染后的正文。
6. 向当前最前方应用发送 `Cmd+V`。
7. 停止，不发送 Enter。

没有 GUI 自动化时，可以使用 `list`、`search`、`get` 和 `pick` 的纯文本流程。

### 2.3 Raycast 模型

Raycast 命令只是薄启动器：

- `scripts/raycast/always.sh` 查找 Python 和仓库相对路径下的 `always.py`，然后运行 `menu`。
- `scripts/raycast/open-json.sh` 用默认编辑器打开 `~/.always/sentences.json`；如果数据库不存在，会先初始化。
- `_always_common.zsh` 集中处理路径和 Python 发现逻辑，所以命令文件不再硬编码 `/opt/homebrew/bin/python3` 或某个具体 checkout 路径。

Raycast 不负责持久化、搜索、渲染、备份或粘贴行为。这些仍然留在核心 Python CLI，确保 Codex、Claude Code、直接 CLI 和 Raycast 不会分叉。

### 2.4 搜索模型

搜索会组合以下字段：

```text
id + title + text + category + language + tags
```

查询和内容都会转为小写。以空格分开的每一个查询词，都必须是组合文本的子字符串。当前没有分词、模糊匹配、相关度评分、词干处理或中文专用切词。

### 2.5 写入模型

新增、修改和删除会读取完整 JSON，在内存里修改 `sentences` 数组，然后：

1. 把现有数据库复制为 `sentences.backup.json`。
2. 把格式化 UTF-8 JSON 写入 `sentences.json.tmp`。
3. 用临时文件原子替换主数据库。

临时文件替换可以避免主文件只写了一半；滚动备份只能保留上一个状态。

### 2.6 变量模型

变量使用 `string.Formatter` 风格，例如 `{file}`。值来自重复的 `--var name=value` 参数或交互提示。渲染发生在选择时，不会修改数据库里的原始模板。

## 三、当前能力边界

### 可以可靠做到的

- 让 Raycast、Codex、Claude Code 和直接 CLI 共用一份本地个人指令库。
- 从 Raycast 全局快捷键启动同一个选择器。
- 从 Raycast 直接打开真实 JSON 数据库，方便查看或编辑。
- 列出和搜索多语言内容。
- 通过 macOS 原生窗口选择。
- 渲染带命名变量的复用模板。
- 粘贴但不自动提交。
- 按 ID 新增、修改、重命名和删除。
- 每次写入前保留一份上一状态备份。
- 数据库缺失时从 sample 初始化。
- 不安装第三方 Python 依赖即可运行。

### 当前做不到的

- 在多台设备之间自动同步。
- 协调多个进程同时写入。
- 加密保存内容。
- 提供完整历史或按条目回滚。
- 模糊搜索、相关度排序或按使用频率排序。
- 保存文件、图片或富文本文档附件。
- 在焦点变化时保证粘贴到正确应用。
- 提供原生图形化常用句编辑器。
- 通过严格 JSON Schema 校验每个字段。
- 提供与 macOS 同级的其他平台选择器和粘贴适配。
- 自动配置 Raycast；Script Directory 和快捷键需要用户手动添加。

## 四、个人常用句与项目记忆的区别

Always 和 ShareMemory 解决的是两类不同上下文：

| 维度 | Always | ShareMemory |
|---|---|---|
| 范围 | 一个人跨项目复用的指令 | 一个项目内的共享 Agent 状态 |
| 存储 | `~/.always/sentences.json` | `<project>/AI_MEMORY/` |
| 典型内容 | 口吻、审查提示、工作流要求 | 决策、任务、经验、交接 |
| 生命周期 | 跨项目、长期 | 跟随具体仓库 |
| 使用方式 | 搜索或选择一句 | 按协议读取、更新项目记忆 |
| Git 属性 | 通常私有且位于仓库外 | 可按项目需要进入版本管理 |

不要把项目决策放进 Always：它会脱离项目并逐渐过期。也不要把个人跨项目偏好放进 ShareMemory：它会污染项目交接。

## 五、短期与长期使用的区别

| 维度 | 小型常用句库 | 大型长期常用句库 |
|---|---|---|
| 选择 | 可以快速扫一眼 | 菜单会越来越难浏览 |
| 搜索 | 简单过滤基本够用 | 缺少模糊排序的问题会变明显 |
| 分类 | 轻量组织即可 | 没有强制分类体系，命名会漂移 |
| 备份 | 一代备份通常够用 | 更早的误改无法恢复 |
| 可迁移性 | 一个文件很方便 | 跨设备同步要自行解决 |
| 维护 | 几乎不用维护 | 重复、过期提示词需要手动整理 |

当前实现最适合经过筛选、数量适中的高价值个人指令。它没有针对数千条片段或长篇文档存储优化。

## 六、Project-wise 优缺点

### 优点

- **纯 JSON**：容易查看、备份、diff 和迁移。
- **单一共享来源**：Raycast、Codex 和 Claude Code 不会各自形成一套提示词。
- **Raycast-first 速度**：全局快捷键能直接打开同一个选择器，不需要 Agent 对话往返。
- **原生快速路径**：常见场景只需一条命令和一次点击。
- **发送权属于用户**：自动粘贴不会变成自动执行。
- **无服务依赖**：不需要账号、网络、API Key 或托管数据库。
- **变量化复用**：`{变量}` 可以把一句话变成一组通用模板。
- **实现很小**：一个 Python 脚本和一个安装脚本，容易审计。

### 缺点

- **macOS 中心化体验**：最佳体验依赖 AppleScript 和 System Events。
- **只有一代滚动备份**：不足以处理深层恢复。
- **没有并发写协议**：后写者覆盖前写者，固定临时路径也会冲突。
- **搜索较基础**：没有模糊匹配、排序、别名或使用统计。
- **校验较宽松**：只要顶层结构有效，部分异常字段仍可能保留下来。
- **依赖源码仓位置**：移动或删除源码仓会让 Agent 软链接和 Raycast Script Directory 同时失效，直到重新指向新位置。
- **明文隐私模型**：只适合用户愿意以本机明文保存的内容。

## 七、风险全清单

### 数据与持久化

1. **只恢复上一代**：每次写入都会覆盖旧备份，只能恢复紧邻的前一个数据库。
2. **备份在同一磁盘**：HOME 或磁盘丢失时，主文件和备份一起丢失。
3. **并发写入**：没有写锁；两个进程可能从同一旧状态读取，再互相覆盖。
4. **共用临时文件名**：并发写入都会使用同一个 `.json.tmp`。
5. **数据结构校验不完整**：只检查顶层对象和 `sentences` 列表，不逐项验证全部字段类型。
6. **手改 JSON**：直接编辑会绕过滚动备份，而且可能让整个数据库无法解析。
7. **强制重置**：`seed --force` 会在保留一代备份后，用 sample 替换真实常用句库。

### 选择与渲染

8. **粘贴依赖焦点**：选择与 `Cmd+V` 之间如果焦点变化，文本可能进入错误应用。
9. **辅助功能权限**：启动脚本的应用没有权限时，System Events 粘贴会失败。
10. **剪贴板暴露**：被选中的文本会留在系统剪贴板，直到被其他内容替换。
11. **模板语法错误**：花括号不匹配或不支持的格式表达式会在渲染时失败。
12. **原生菜单只显示标题**：看不到分类和预览；标题重复或过于模糊时难以选择。
13. **基础 AND 搜索**：换一种说法可能完全搜不到概念上相关的常用句。

### 身份与组织

14. **ASCII ID 标准化**：自动 ID 会移除非 ASCII 字符；纯中文标题必须显式提供 ID。
15. **分类体系不强制**：category 和 tag 容易出现 `writing`、`write`、`写作` 等漂移。
16. **重命名引用失效**：修改 ID 可能破坏使用旧 ID 的别名、脚本或文档。
17. **标签是替换语义**：编辑时传入 `--tag` 会替换整个标签列表，不是追加。

### 安装与分发

18. **移动源码仓**：软链接目标和 Raycast Script Directory 都会在仓库移动后失效。
19. **现有目录冲突**：安装器拒绝覆盖非软链接目录，安全但需要用户手动处理。
20. **平台不对称**：安装与文本 CLI 可在其他平台使用，但 `menu` 和 `--paste` 是 macOS 专属。
21. **当前没有自动化测试套件**：在补齐测试和 CI 前，主要依赖语法检查和 smoke test。
22. **未来数据迁移**：文件声明了 `version: 1`，但当前还没有迁移框架。
23. **Raycast PATH 差异**：Raycast 可能不会继承交互式 shell 的 PATH，所以脚本必须防御式查找 Python。
24. **Raycast 设置漂移**：脚本改名、目录移动或快捷键变化，都可能让文档与本机 Raycast 设置不同步。

### 隐私

25. **明文存储**：任何能访问用户账号的人都能读取保存的提示词。
26. **误存秘密**：用户可能因为复用方便，把凭据写进常用句，但 Always 不是密钥库。
27. **屏幕和历史暴露**：打印出的句子可能进入终端回滚、截图、自动化日志或 Agent 对话记录。

## 八、谁适合，谁不适合

**强烈适合**：

- 想用 Raycast 全局快捷键调用常用提示词的 macOS 用户。
- 使用 Codex 和 / 或 Claude Code，并希望 Agent 对话里也能访问同一份库的人。
- 有一组会反复使用的提示词和工作偏好的人。
- 看重一键原生选择器的 macOS 用户。
- 喜欢透明本地文件，不想使用托管片段服务的人。
- 写入操作按顺序进行的单人工作流。

**谨慎使用或选择其他工具**：

- 需要团队共享、权限管理和远程同步的组织。
- 需要保存秘密或受监管数据的场景。
- 需要模糊搜索、排序、分析或富文本编辑的大型库。
- 多个进程可能同时写入的自动化流程。
- 必须在非 macOS 平台获得完整原生选择体验的人。

## 九、维护者快速接手（5 分钟）

按以下顺序检查：

1. 仓库初始化 git 后，先看 `git status --short` 和 `git diff --stat`。
2. 读 `skills/always/SKILL.md`，确认 Agent 触发和执行规则。
3. 读 `scripts/raycast/`，确认 Raycast 启动器行为和快捷键设置说明。
4. 读 `skills/always/scripts/always.py`，确认 CLI、持久化、搜索、渲染和粘贴行为。
5. 读 `skills/always/assets/sentences.sample.json`，确认第一次初始化内容。
6. 读 `scripts/install.sh`，确认分发路径和冲突处理。
7. 读 `README.md` 与 `README.zh.md`，确认对外承诺。
8. 读本中英详解对，确认架构理由与已知风险。

| 想确认什么 | 唯一真源 |
|---|---|
| Agent 触发与预期行为 | `skills/always/SKILL.md` |
| Raycast 命令与设置 | `scripts/raycast/README.md` + `scripts/raycast/*.sh` |
| CLI 参数和运行时行为 | `skills/always/scripts/always.py` |
| 初始默认内容 | `skills/always/assets/sentences.sample.json` |
| 安装路径与软链接 | `scripts/install.sh` |
| 用户真实数据 | `~/.always/sentences.json` |
| 对外使用契约 | `README.md` + `README.zh.md` |
| 架构、取舍与风险 | `PROJECT_DETAILS.md` + `项目详解.md` |

发布候选前运行：

```bash
python3 -m py_compile skills/always/scripts/always.py
bash -n scripts/install.sh
zsh -n scripts/raycast/always.sh
zsh -n scripts/raycast/open-json.sh
python3 skills/always/scripts/always.py --help
```

随后使用隔离 HOME 做一次 smoke test，覆盖 `seed`、`list`、`search`、`get`、`add`、`edit` 和 `delete`。原生选择器和 Raycast 快捷键依赖焦点、Raycast 设置和辅助功能状态，必须在 macOS 上人工验证。

## 附录 A：当前设计决策

### A.1 数据库位于仓库之外

个人内容放在 `~/.always/`，不进入 Skill 源码仓。这样代码更新与用户数据彼此独立，也能让两个 Agent 指向同一份库。代价是只克隆 GitHub 仓库不会恢复个人内容。

### A.2 两个 Agent 软链接到同一份 Skill

安装器把两个发现路径都链接到 `skills/always/`，避免代码漂移并让更新立即生效。复制式安装在移动源码后更独立，但需要显式同步两份副本。

### A.2.1 Raycast 只做启动器，不做核心

Raycast 是最快的用户入口，但不应该承载业务逻辑。Script Commands 保持很薄，只调用仓库内的 Python CLI。这样 Raycast、Codex、Claude Code 和直接终端使用都走同一套实现，不会在 shell 脚本里重复实现搜索、渲染、备份和粘贴逻辑。

### A.3 粘贴但不按 Enter

Always 把选中的句子视为可编辑的用户输入。它有意停在粘贴之后，让用户在执行前检查变量、上下文和目标位置。

### A.4 滚动备份加原子替换

当前持久化模型在不引入数据库的前提下提供最低安全线：保存旧文件、完整写入临时 JSON、再替换主文件。它不宣称支持并发安全或长期历史。

### A.5 原生选择器作为默认路径

对于小型常用句库，原生菜单能减少 Agent 对话和视觉扫描成本。Raycast 让这个选择器可以全局调用；纯文本命令继续服务于查看、自动化和无法控制 GUI 的环境。

### A.6 双语 GitHub 文档基线

公开文档沿用 ShareMemory 的同一品牌结构：居中的双语入口、明确的问题陈述、诚实的能力边界、中英文详解对，以及 `yc星辰` 名下的 MIT License。在仓库真正拥有对应自动化之前，不虚构 CI badge 或测试覆盖声明。
